# Crypto-Shredding on BigQuery

[Crypto-shredding](https://en.wikipedia.org/wiki/Crypto-shredding) is a technique where data is deleted by destroying the cryptographic keys used to encrypt it. This makes the encrypted data permanently unrecoverable.

This method offers efficiency because it involves removing a single key rather than performing updates on large and potentially immutable tables. In addition, it provides comprehensive coverage; once the key is removed, the associated data becomes inaccessible across all environments, including logs and backups.

For BigQuery another specific advantage is that this way large partitioned tables that are benefiting from [Long Term Storage](https://cloud.google.com/blog/topics/developers-practitioners/bigquery-explained-storage-overview) won’t be impacted (as they don’t get updated).

## Implementation on BigQuery

The main idea of crypto-shredding is to add a specific key for every user in a key registry table and use that key to encrypt all of the user’s (PII) data. If we need to access the user data, we then use the same key to decrypt it. Once we need to delete the user data, all we need to do is to remove their key from the key registry table.

There are two methods of managing the keys on BigQuery; we can use and store raw keys in BigQuery or we can use [KMS](https://docs.cloud.google.com/kms/docs/key-management-service) and wrapped keys. Each of these options have their advantages and disadvantages.

In this repository we'll demonstrated both of these appraoches. The raw keys approach is presented first as it is the most cost-effective and practical approach for crypto-shredding. Wrapped keys require involvement of KMS for every encryption and decryption operation; although this works fine for granular operations, it poses some challenges when we need to process large amounts of data in bulk. This will involve including external processes/functions to handle the encryption/decryption in bulk (which comes with additional costs and complexity). Nevertheless, if you need encryption beyond crypto-shredding (with additional requirements such as detailed auditing, key rotation, adhering to specific compliance frameworks etc.) then KMS with wrapped keys would be the recommended alternative.

> [!NOTE]  
> For crypto-shredding to be an effective method, the key deletion has to be irreversible. So, any potential copies, backups or snapshots of the key registry table would render this approach useless.

## Setup

We’ll start simple, and assume that the source user data (not-encrypted) is already in BigQuery in a temporary (with TTL) table so that we can do the encryption in bulk. Alternatively, we could do the encryption during the ingestion pipelines (potentially outside of BigQuery).

In this scenario, we assume a large transaction table is going to be our target. We will append the encrypted user data to it using their existing keys, while provisioning new keys for any new user.

The DDL for the user transaction table in our example will look like this:

```sql
CREATE OR REPLACE TABLE crypto_shredding_demo.user_transactions (
  transaction_id STRING,
  user_id INT64,
  transaction_timestamp TIMESTAMP,
  -- Non-PII columns (unencrypted plaintext)
  amount NUMERIC,
  currency STRING,  
  -- PII columns (encrypted with AEAD, stored as BYTES)
  user_email BYTES
);
```

We’ll encrypt only the PII columns and leave others as is. These encrypted columns must be of type `BYTES` because the encryption function always outputs a byte array, regardless of original data type.

> [!NOTE]  
> For the sake of simplicity we're keeping the number of PII columns limited, but the same approach would be applied if you had more PII columns. Note that if you have columns that are not PII by themselves but could lead to identifying an individual when combined with other columns, they would be candidates for encryption too.

The key registry table schema is even simpler. It maps each `user_id` to their unique cryptographically secure keyset:

```sql
CREATE OR REPLACE TABLE crypto_shredding_demo.user_keys (
  user_id INT64,
  user_key BYTES
);
```

> [!NOTE]  
> Keep in mind that if you use encrypted columns as (part of) your clustering, you might not get the typical performance improvements from clustering. This is related to the fact that random-like data (as generated by encryption) typically is spread over a long range, which causes storage blocks to have overlapping ranges. So even if the data is sorted, the min/max values will be similar for multiple blocks, making the pruning less effective.

## Raw Keys on BigQuery

When we use the the raw keys in BigQuery, we generate and store keysets directly inside a BigQuery table (`crypto_shredding_demo.user_keys`) as raw `BYTES`. We will use *deterministic encryption* (`DETERMINISTIC_AEAD_AES_SIV_CMAC_256`) because it produces identical ciphertext for the same plaintext and key. This allows us to perform exact matches, `GROUP BY`, and joins on the encrypted columns.

### Staging Table Schema

Assume we have a temporary/staging table containing unencrypted raw transactions:

```sql
CREATE OR REPLACE TABLE crypto_shredding_demo.staging_transactions (
  transaction_id STRING,
  user_id INT64,
  transaction_timestamp TIMESTAMP,
  amount NUMERIC,
  currency STRING,
  user_email STRING  -- not encrypted
);
```

### Creating the Keys

To ensure that every user present in the staging table has a corresponding keyset in the `user_keys` table before we process the transactions, we run a `MERGE` statement:

```sql
MERGE crypto_shredding_demo.user_keys AS target
USING (SELECT DISTINCT user_id FROM crypto_shredding_demo.staging_transactions) AS source
ON target.user_id = source.user_id
WHEN NOT MATCHED THEN
  INSERT (user_id, user_key)
  VALUES (
    source.user_id, 
    KEYS.NEW_KEYSET('DETERMINISTIC_AEAD_AES_SIV_CMAC_256')
  );
```

> [!NOTE]  
> `KEYS.NEW_KEYSET` evaluates at row-level when executing an `INSERT`, meaning a distinct, unique 512-bit keyset is generated for each new `user_id`.

### Inserting and Encrypting Data

Now we can insert transactions from the staging table into the target table, using `DETERMINISTIC_ENCRYPT` to encrypt the PII columns:

```sql
INSERT INTO crypto_shredding_demo.user_transactions (
  transaction_id,
  user_id,
  transaction_timestamp,
  amount,
  currency,
  user_email
)
SELECT
  s.transaction_id,
  s.user_id,
  s.transaction_timestamp,
  s.amount,
  s.currency,
  -- Encrypt PII deterministically using the user's unique keyset
  DETERMINISTIC_ENCRYPT(k.user_key, s.user_email, CAST(s.user_id AS STRING)) AS user_email,
FROM crypto_shredding_demo.staging_transactions AS s
JOIN crypto_shredding_demo.user_keys AS k
ON s.user_id = k.user_id;
```

> [!NOTE]  
> Passing `s.user_id` as the third argument (`additional_data`) binds the ciphertext to that specific `user_id`. This is a security best practice that prevents "ciphertext-substitution" attacks, where an attacker copies encrypted data from User A to User B. Decryption will fail unless the correct `user_id` is supplied.

### Querying and Decrypting Data

To read the plaintext transactions, we join with the key registry and decrypt the PII columns using `DETERMINISTIC_DECRYPT_STRING`:

```sql
SELECT
  t.transaction_id,
  t.user_id,
  t.transaction_timestamp,
  t.amount,
  t.currency,
  t.merchant_category,
  DETERMINISTIC_DECRYPT_STRING(k.user_key, t.user_email, CAST(t.user_id AS STRING)) AS user_email
FROM crypto_shredding_demo.user_transactions AS t
JOIN crypto_shredding_demo.user_keys AS k
ON t.user_id = k.user_id;
```

If a user exercises their "right to be forgotten", we simply execute:

```sql
DELETE FROM crypto_shredding_demo.user_keys
WHERE user_id = <target_user_id>;
```

Once the key is deleted, subsequent decryptions for that user will fail or return `NULL` (since the key is gone), effectively shredding the transactional history's PII data without altering the large `user_transactions` table.

### Additional Considerations

Although the goal of the encryption in this example is to enable crypto-shredding, if additional security is needed so that the encryption keys are only available to a limited audience, you could apply [Column Level Security](https://docs.cloud.google.com/bigquery/docs/column-level-security-intro) to the `user_key` column in the key registry table. In that case decryption would only be possible if the user/service account is granted the appropriate permissions.

One of the advantages of this approach is also its costs. Generating and storing the keys in BigQuery only results on standard BigQuery storage and processing costs.

## KMS and Wrapped Keys

For workloads requiring high compliance, detailed access auditing, and strong separation of duties, storing raw symmetric keysets in BigQuery (even with column-level security) might not be sufficient.

This KMS Wrapped Keys approach implements *Envelope Encryption*:

1. A master *Key Encryption Key (KEK)* is created and managed inside Google Cloud KMS.
2. For each user, a local symmetric keyset is generated, this is the *Data Encryption Key (DEK)*.
3. The DEK is encrypted (wrapped) with the KMS KEK, and only this *wrapped keyset* is stored in the BigQuery `user_keys` registry table.
4. When performing encryption or decryption, the wrapped keyset is sent to a function where it is unwrapped using Cloud KMS and used locally in-memory for deterministic encryption/decryption via a Python library.

This ensures that cleartext key material never resides in storage, and offline exfiltration of the BigQuery tables is completely harmless.

### Creating the Wrapped Keys with KMS

Just like raw keys can be generated natively in BigQuery using `KEYS.NEW_KEYSET`, KMS-wrapped keys can also be provisioned natively in BigQuery SQL using `KEYS.NEW_WRAPPED_KEYSET`:

To ensure that every user present in the staging table has a corresponding keyset in the `user_keys` table before we process the transactions, we run a `MERGE` statement:

```sql
MERGE crypto_shredding_demo.user_keys AS target
USING (SELECT DISTINCT user_id FROM crypto_shredding_demo.staging_transactions) AS source
ON target.user_id = source.user_id
WHEN NOT MATCHED THEN
  INSERT (user_id, user_key)
  VALUES (
    source.user_id, 
    KEYS.NEW_WRAPPED_KEYSET( -- this is the main difference compared to raw keys approach
      'gcp-kms://projects/your-project/locations/your-location/keyRings/your-key-ring/cryptoKeys/your-key',
      'DETERMINISTIC_AEAD_AES_SIV_CMAC_256'
    )
  );
```

This generates a new secure keyset, wraps (encrypts) it with Cloud KMS, and returns it as `BYTES` natively within standard SQL.

### Batch Encryption/Decryption with Wrapped Keys

One of the challenges of working with wrapped keys is that BigQuery's native `KEYS.KEYSET_CHAIN` function requires the wrapped keyset argument to be a *constant or literal string*. Since we are implementing a crypto-shredding model where each user has their own unique keyset stored in a table column (`user_keys.user_key`), we cannot pass this dynamic column value directly to `KEYS.KEYSET_CHAIN` at row-level, making it hard to do encryption/decryption in bulk.

In order to address that we can utilize a hybrid approach where we create the wrapped keys in BigQuery (as shown in the previous section) and leave the encryption and decryption in bulk to another process that can handle batches dynamically. BigQuery supports [Remote Functions](https://docs.cloud.google.com/bigquery/docs/remote-functions) which allow us to implement a user-defined function in, for example Python, and hosted on Cloud Run.

So, for batch encryption and decryption we will have two fully generic functions:

* `encrypt_string_remote(wrapped_key, plaintext, user_id)`:
  Takes the wrapped keyset from `user_keys`, any plaintext string column (such as email, phone, name, or other PII), and the `user_id` (used as AAD) and returns base64-encoded encrypted BYTES.
* `decrypt_string_remote(wrapped_key, ciphertext, user_id)`:
  Takes the wrapped keyset, the encrypted ciphertext, and the `user_id` and returns the decrypted plaintext string.

Once these functions are deployed and registered in BigQuery, we can then run the encryption:

```sql
INSERT INTO crypto_shredding_demo.user_transactions (
  transaction_id,
  user_id,
  transaction_timestamp,
  amount,
  currency,
  user_email
)
SELECT
  s.transaction_id,
  s.user_id,
  s.transaction_timestamp,
  s.amount,
  s.currency,
  -- Call our Cloud Run Remote Function to encrypt using the wrapped keyset
  crypto_shredding_demo.encrypt_string_remote(k.user_key, s.user_email, s.user_id) AS user_email
FROM crypto_shredding_demo.staging_transactions AS s
JOIN crypto_shredding_demo.user_keys AS k
ON s.user_id = k.user_id;
```

And similarly the decryption would look like this:

```sql
SELECT
  t.transaction_id,
  t.user_id,
  t.transaction_timestamp,
  t.amount,
  t.currency,
  CASE
    -- If the key in the registry is deleted, the user has been crypto-shredded
    WHEN k.user_key IS NULL THEN 'DELETED'
    -- Otherwise, invoke the remote function to unwrap keyset and decrypt
    ELSE crypto_shredding_demo.decrypt_string_remote(k.user_key, t.user_email, t.user_id)
  END AS decrypted_email
FROM crypto_shredding_demo.user_transactions AS t
LEFT JOIN crypto_shredding_demo.user_keys AS k
ON t.user_id = k.user_id;
```

### Additional Considerations for Wrapped Keys

Although we have implemented some basic caching in the provided remote functions, this approach basically requires KMS to encrypt and decrypt every field. This can be a costly operation if you have to process (encrypt/decrypt) large amounts of data frequently.

## Demo Scripts

We have included a sample Terraform script that sets up a few things to demonstrate the approach described in this repository. When you run it, it will create a staging table and populate it with random data. The queries to do key creation, encryption/decryption and key deletion are provide as stored procedures for the ease of use, but keep in mind that they don't have to be stored procedures.

All BigQuery Stored Procedures and Remote Functions are automatically provisioned. See the Terraform output (`demo_instructions` and `kms_demo_instructions`) for the exact SQL commands and CLI instructions to demonstrate both approaches.

> [!NOTE]  
> If you want to try out both approaches makes sure to truncate the `user_keys` and `user_transactions` tables as the generated keys are not compatible with each other.
